# Phase 5: フレーミング統合（仮想カメラ ROI）

> 本ドキュメントは CLAUDE.md プリンシパル8に従い、Phase 5 の技術選定・設計判断・
> 評価指標・他競技展開を記録する。CLAUDE.md は本書を参照するに留める。

## 1. 目的とスコープ

ボール位置（Phase 3）と field 選手集団（Phase 4 知覚コア）から「注目領域 ROI」を
算出し、放送的に滑らかな仮想カメラ座標を出力する。

**スコープ（ユーザー合意 2026-06）**:
- 成果物は **ROI 座標の算出＋既存オーバーレイ可視化／クロップ確認**まで。
  実運用のクロップ・レンダリング配信は Phase 5.x。
- 調律方針は **滑らかさ優先（放送的）**。多少追従が遅れてもカメラ移動を滑らかに。
- PTZ の理想優先度は **パン → ズーム → チルト（ほぼ無し）**。ズームは決定機・
  サーブ等の局面で行う（局面シグナルが揃うまで仕組みのみ用意）。

## 2. 放送カメラワークのセオリー調査（プリンシパル4）

実装前に先行研究・放送実践を調査した（カットアンドトライ禁止）。要点:

### 2.1 プロのフレーミング原則
- **リードルーム**: 動く被写体の進行方向に空間を空ける（詰まり感の回避）。
- **三分割法・ヘッドルーム**: 被写体を画面中央に貼り付けない。
- **「ズームしながら撮るな」**: ズーム中の映像はブレ・歪みで視聴者が不快。
  ズームは局面の切れ目でゆっくり行う。
- **PTZ 速度制限**: 現場機材は最大速度・加減速カーブ（K_ACC/K_DEACC）を持ち、
  急発進・急停止を避ける。減速定数 ≥ 加速定数。

### 2.2 先行研究
- **Ariki 2006 (IEEE ISM)** — 超高解像度映像から仮想クロップでパン/ズーム。
  ゲーム状況（PK/FK 等）を認識し**状況別にカメラワークを切替**。本プロジェクトと
  同型のアーキテクチャ。→ 教訓: **カメラ挙動はゲーム状態で切り替える**。
- **OHMM (Chen & Little, CVIU)** — 選手位置→カメラ角度の 1 対多の不確かさを
  Overlapped Hidden Markov Model で時系列平滑化。単純 EMA より自然。
  → 教訓: **1 次フィルタ（EMA）では放送的な滑らかさに届かない**。
- **Pixellot / volleyball_analytics** — ゲーム状態を SERVICE / PLAY / NO-PLAY に
  分類し、ラリーは wide、決定機はズームインと挙動を変える。
- **Unity SmoothDamp / Game Programming Gems 4** — **臨界制動スプリング**。
  速度を状態に持つ 2 次フィルタで、オーバーシュートせず最短収束。ease-in/out が
  自然。`smooth_time` 1 つで追従速度を決められる。

調査の出典は本フェーズ会話ログおよび上記公開資料。

## 3. 旧実装（速度制限つき EMA）の失敗と原因

実映像評価（2026-06）で **NG**:
1. **アスペクト比が毎フレーム変化**して映像がグニャグニャ伸縮（超横長素材 2160×650）。
   原因: `_clamp_roi` が境界クランプ後にアスペクト比を再適用していなかった。
2. **パンが速すぎてカクつき、左右にスウェー**。
   原因: ① ボール重み過大（0.75）でボールの左右動に追従しすぎ。② EMA は速度を
   持たないため目標が動き続けると遅れて追い、速度上限クランプに当たると跳ねる。

→ EMA ベースは構造的限界。**臨界制動スプリングへ全面置換**し、**ゲーム状態認識**を
導入する設計に再構築した。

## 4. 設計（採用）

### 4.1 臨界制動スプリングで平滑化（`_smooth_damp`）
カメラ中心（pan）とサイズ（zoom）を、速度を状態に持つ 2 次フィルタで追従:

```
omega = 2 / smooth_time
exp   = 1 / (1 + x + 0.48 x² + 0.235 x³),  x = omega·dt
out   = target + (change + temp)·exp,  vel を更新（符号反転で target にスナップ）
```

- オーバーシュートせず最短収束、目標が止まれば速度も減衰して滑らかに停止。
- pan/zoom を別 `smooth_time` で調律（既定 pan=0.9s, zoom=1.6s。zoom を緩く=
  ハンチング抑制）。pan には最大速度上限（`_MAX_PAN_SPEED_FRAC=0.6`/秒）。
- テスト `test_smooth_damp_settles_without_overshoot` で単調収束を保証。

### 4.2 ゲーム状態でカメラ挙動を切替（`alt_pix/game_state.py`）
`GameStateEstimator` が 3 状態を推定（ヒステリシス付き）:

| 状態 | 判定（ヒューリスティック） | カメラ挙動 | 目標 zoom |
|---|---|---|---|
| `RALLY` | ボールがコート上で active | 選手集団アンカー＋ボール補正で追従 | 0.72（中ズーム） |
| `SERVICE` | ボール足元射影がエンドライン ±2m | **サーバー（assume_server）に寄せる中ズーム** | 0.58（サーバー判明時）／0.92（不明時 wide） |
| `NO_PLAY` | participation の on-court 人数 EMA が低い | パン凍結・wide 保持（出入りを追わない） | 0.98 |

**サーブ時のサーバー追従（ユーザー要望 2026-06）**: サーブ局面でコート全体を
ルーズに撮ると違和感が強い。`ParticipationTracker.server_track_id()` が
エンドライン外へ出た選手（`assume_server`）を返し、framing はその足元を `focus_xy`
として受け取り、ルーズな wide ではなくサーバー中心の中ズーム（zoom 0.58）に寄せる。
サーバーが特定できないフレームのみ従来の wide（0.92）にフォールバックする。
SERVICE はボールがエンドライン付近にある短い局面でのみ発火するため、サーバーへの
寄りはサーブ前後の数秒に限定され、ラリーに入ると RALLY へ戻ってボール追従に移る。

zoom レベル = ROI 高 / フレーム高。状態遷移は `_SWITCH_HYSTERESIS=8` フレーム連続
支持で確定（NO_PLAY は participation が既に EMA 平滑なので即時）。

**注意（プリンシパル4・8）**: SERVICE/NO_PLAY 判定は現時点ではヒューリスティック。
確実なサーブ・デッドボール検出には action recognition（Phase 6）が要る。framing は
状態 enum にのみ依存するので、将来推定器を差し替えても影響しない。

### 4.2.5 撮り味プロファイル（normal / dynamic）
有人カメラの実映像はより**ダイナミック**（寄りが強い・ボール追従が速い・リード
ルームが大きい）。これを `FramingProfile` として束ね、`--framing-style` で切替:

| プロファイル | 性格 | 主な違い |
|---|---|---|
| `normal`（既定） | 放送的・安定。揺れを抑えゆったり追う | pan 0.9s / zoom 1.6s / ball 0.45 / rally zoom 0.72 / service zoom 0.58 |
| `dynamic` | 有人カメラ的。寄り強め・機敏 | pan 0.55s / zoom 1.1s / ball 0.68 / rally zoom 0.55 / service zoom 0.46 / lead 0.30 / 決定機 zoom 0.38 / min zoom 0.28 |

両者ともゲーム状態認識・サーバー端寄せ（SERVICE ではサーバーをフレーム端に置き
コート側へ空間を確保）・臨界制動スプリングの土台は共有し、時定数・ボール重み・
zoom 目標・リードルームの**ダイヤルだけ**を差し替える。プロファイルは
`alt_pix/framing.py` の `PROFILES` に定義（追加も容易）。

### 4.3 リードルーム
ボール進行方向（水平主体・垂直 ×0.3）へ中心を `_LEAD_GAIN=0.18` でオフセット
（上限 `_LEAD_MAX_FRAC=0.12`）。バレーは pan 優先のため水平を主とする。

### 4.4 ズームイン演出のフック（プレースホルダ）
スパイク・決定機での寄りは action recognition が要るため、現時点では
`compute(..., highlight=True)` でズームイン→保持→戻しのエンベロープ
（`_HIGHLIGHT_RATE` で増減）だけ実装。トリガ信号（Phase 6）が来たら効く。
既定 `highlight=False` なので現状の挙動には影響しない。

## 5. 候補比較（プリンシパル4）

| 平滑化手法 | 速度状態 | 特性 | 採否 |
|---|---|---|---|
| EMA（旧） | なし | 遅延追従＋停止時オーバーシュート | 不採用（NG の主因） |
| 速度制限つき EMA（旧改良） | なし | 上限で跳ねてカクつく | 不採用 |
| **臨界制動スプリング（採用）** | あり | 単調収束・自然な ease-in/out・1 パラメータ | ★採用 |
| OHMM / LQR | あり | 最も自然だが実装・チューニング重い・遅延前提 | 将来（バッチ時） |
| 学習ベース（放送模倣） | — | 人間的画作り・教師データと商用重み制約 | Phase 6 以降 |

カメラワーク方針も「状態機械での明示遷移」vs「状態別パラメータ＋連続平滑化」を比較し、
後者（状態は zoom 目標とパン挙動の切替に使い、動き自体はスプリングで連続）を採用。
状態境界での跳ねを避けられる。

## 6. 評価（プリンシパル2: 1フェーズ1評価）

`scripts/eval_framing.py out.jsonl --frame-w W` で JSONL から再現可能に算出:

| 指標 | 意味 | 望ましい方向 |
|---|---|---|
| `ball_in_roi%` | 可視ボールが ROI 内にある割合 | 高 |
| `field_in_roi%` | field 選手 bbox 面積が ROI に入る平均割合 | 高 |
| `pan_jitter_px` | ROI 中心の 2 階差分 RMS（加速度＝ジッター） | **低** |
| `zoom_jitter` | ROI 幅の 2 階差分 RMS / 平均幅（伸縮往復） | **低** |

`scripts/render_framed.py` で ROI を実際に切り出した仮想カメラ映像を書き出し、
**出力映像そのもの**で滑らかさ・アスペクト・状態遷移を目視評価する（`--compare` で
ソースと縦並べ）。

`tests/test_framing.py`（8 件）・`tests/test_game_state.py`（5 件）で GPU 不要に
挙動を保証: スプリング単調収束、16:9 維持、低ジャーク pan、NO_PLAY の wide 凍結、
SERVICE の wide 化、ボール瞬断の非跳躍、状態ヒステリシス。

## 7. 旧実装からの改善履歴

1. EMA → 臨界制動スプリングへ全面置換（カクつき・スウェーの根治）。
2. `game_active`(bool) → 3 状態（RALLY/SERVICE/NO_PLAY）に拡張。
3. アスペクト比再適用バグ修正（横長素材の歪み解消）。
4. ボール重み 0.75→0.45（選手集団アンカー化でスウェー抑制）。
5. リードルーム・ズーム演出フックを追加。

## 8. 他競技への展開

ターゲット算出と状態認識は競技非依存に近い。調律点:

| 競技特性 | 調律 |
|---|---|
| ボールが速い（テニス・卓球） | `_LEAD_GAIN`↑, `_W_BALL_VISIBLE`↑, pan `smooth_time`↓ |
| 集団が広く動く（サッカー・ハンドボール） | `_ZOOM_RALLY`↑（wide 寄り）, `_MIN_ZOOM`↓ |
| ネット無し競技 | SERVICE 判定を競技の局面定義に差し替え |
| ボール検出が不安定 | `_W_BALL_PREDICTED`↓で選手集団主体に |

action recognition（Phase 6）導入後は、`GameStateEstimator` を学習ベースの局面
分類に差し替え、`highlight` をスパイク/ブロック検出に結線する。
