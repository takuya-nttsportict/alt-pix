# Phase 5: フレーミング統合（仮想カメラ ROI）

> 本ドキュメントは CLAUDE.md プリンシパル8に従い、Phase 5 の技術選定・設計判断・
> 評価指標・他競技展開を記録する。CLAUDE.md は本書を参照するに留める。

## 1. 目的とスコープ

ボール位置（Phase 3）と field 選手集団（Phase 4 知覚コア）から「注目領域 ROI」を
算出し、滑らかにカメラ追従する仮想カメラ座標を出力する。

**今フェーズのスコープ（ユーザー合意 2026-06）**:
- 成果物は **ROI 座標の算出＋既存オーバーレイ可視化**まで。実際のクロップ／
  レンダリング出力は Phase 5.x へ分離する。
- 調律の方針は **滑らかさ優先（放送的）**。多少追従が遅れてもカメラ移動を滑らかに。

入出力:
- 入力: `BallState`（visible / predicted / lost）、field 選手 `Track` 群、
  `participation.game_active`（タイムアウト／セット間検出）。
- 出力: `framing_roi`（x, y, w, h）を JSONL に記録。アスペクト比 16:9 固定。

## 2. 旧実装の課題

旧 `framing.py` は次の素朴なロジックだった:
- `auto`: ボール可視 → ボールタイト枠、不可視 → 全選手 wide のハード切替。
- 単一 EMA（α=0.15）で中心＋サイズをまとめて平滑化。

観測された問題:

| # | 課題 | 原因 |
|---|---|---|
| 1 | ボール瞬断のたびに wide↔ball で画が跳ぶ | 可視/不可視のハード切替 |
| 2 | タイムアウト中もカメラが選手の出入りを追う | game_active 未使用 |
| 3 | predicted ボール（Kalman 補間）を検出と同等に信頼 | 確度の区別なし |
| 4 | ズームの伸縮ハンチング、pan のジッター | pan/zoom 同一 EMA・速度制限なし |
| 5 | 高速ボールにカメラが後追い | 先読みなし |

## 3. 設計（採用）

### 3.1 ターゲット ROI: 加重合成（ハード切替の廃止）

中心をボールと選手集団重心の加重平均にする:

```
center = w_ball · (ball + lookahead) + (1 - w_ball) · players_centroid
size   = w_ball · ball_tight_box     + (1 - w_ball) · players_bbox
```

`w_ball` をボール確度で連続的に変える:

| ボール状態 | w_ball | 意図 |
|---|---|---|
| visible（検出） | 0.75 | ボール主体（プレーの焦点） |
| predicted（Kalman 補間） | 0.45 | 選手集団へ寄せ始める（補間の不確かさを織り込む） |
| lost（喪失） | 0.00 | 選手集団のみ（暴れない） |

ボールが 1 フレーム消えても重みが滑らかに移るだけで画は跳ばない（課題 1・3 解決）。

### 3.2 pan / zoom 分離平滑化＋速度制限＋デッドゾーン

中心（pan）とサイズ（zoom）を別 EMA・別速度上限で平滑化（課題 4 解決）:

| パラメータ | 既定 | 意味 |
|---|---|---|
| `pan_alpha` | 0.12 | 中心 EMA 係数（低いほど滑らか） |
| `zoom_alpha` | 0.06 | サイズ EMA 係数（pan より緩く＝伸縮往復を抑制） |
| `max_pan_frac` | 0.04 | 1 フレームの中心移動上限（フレーム幅比、≈77px@1920） |
| `max_zoom_rate` | 0.03 | 1 フレームのサイズ変化上限（現サイズ比） |
| `deadzone_frac` | 0.03 | この範囲内のターゲット中心移動は無視（≈58px@1920） |

デッドゾーンで微小なボール揺れに反応しない（ジッター除去）。速度制限でターゲットが
急に飛んでもカメラ振りが一定以下に保たれる。

### 3.3 先読み（look-ahead）

直近 2 フレームの**可視**ボール速度ベクトルを `lookahead_frames=3` ぶん中心に
加算（上限 `LOOKAHEAD_MAX_FRAC=0.15`）。滑らかさ優先のため控えめ。predicted /
lost フレームでは先読みを無効化（補間速度は信頼しない）（課題 5 緩和）。

### 3.4 ゲーム状態ゲーティング

`game_active=False`（`participation` がコート上人数 EMA の激減で検出）なら、
ボールを無視し選手集団中心の wide（`_PAUSE_ROI_FRAC=0.85`）に緩く保持する。
タイムアウト・セット間で選手がコート外へ歩いてもカメラが振り回されない（課題 2 解決）。

## 4. 候補比較（プリンシパル4）

| 手段 | 長所 | 短所 | 採否 |
|---|---|---|---|
| **加重合成＋速度制限 EMA（採用）** | 依存ゼロ・決定的・調律容易・瞬断に頑健 | 物理的に最適ではない（経験則パラメータ） | ★採用 |
| 状態機械（ball-follow / rally-wide を明示遷移） | 各状態の意図が明快 | 遷移境界でやはり跳ねる／ヒステリシス調律が増える | 不採用（課題1再発） |
| Kalman / LQR で仮想カメラを最適制御 | 理論的に滑らか | チューニング・実装コスト大、ボール検出ノイズに敏感 | 将来検討 |
| 学習ベース（放送カメラ模倣） | 人間的な画作り | 教師データ（プロのカメラワーク）が必要・商用重み制約 | Phase 6 以降で検討 |

滑らかさ優先・依存最小・調律しやすさから加重合成＋速度制限 EMA を採用。将来、実
カメラワークのデータが得られれば LQR/学習ベースへ差し替え可能なよう、ターゲット
算出（`_target`）と平滑化（`_smooth_*`）を分離してある。

## 5. 評価（プリンシパル2: 1フェーズ1評価）

`scripts/eval_framing.py out.jsonl --frame-w W` で JSONL から再現可能に算出:

| 指標 | 意味 | 望ましい方向 |
|---|---|---|
| `ball_in_roi%` | 可視ボールが ROI 内にある割合 | 高 |
| `field_in_roi%` | field 選手 bbox 面積が ROI に入る平均割合 | 高 |
| `pan_jitter_px` | ROI 中心の 2 階差分 RMS（＝加速度＝ジッター） | **低** |
| `pan_speed_px` | ROI 中心の 1 階差分平均（pan 速度、参考） | — |
| `zoom_jitter` | ROI 幅の 2 階差分 RMS / 平均幅（伸縮往復） | **低** |
| `roi_off_frac` | ROI がフレーム端にクランプした割合（参考） | 低（過寄り検出） |

旧実装と新実装で同一 JSONL 入力素材を流し、`ball_in_roi%` を維持しつつ
`pan_jitter` / `zoom_jitter` が下がることを確認する（before/after 比較）。

`tests/test_framing.py`（GPU 不要）で挙動を保証:
- pan / zoom の 1 フレーム移動量が上限以下
- 1 フレームのボール喪失で画が跳ばない
- デッドゾーン内の微小揺れでカメラが動かない
- predicted ボールが検出ボールより選手集団へ寄る
- ゲーム停止で wide 保持しボールを追わない

## 6. 他競技への展開

ターゲット算出は競技非依存（ボール＋参加選手集団）。平滑化パラメータのみ調律する:

| 競技特性 | 調律 |
|---|---|
| ボールが速い競技（テニス・卓球） | `lookahead_frames` を増やし `w_ball` を上げる |
| 集団が広く動く競技（サッカー・ハンドボール） | `_WIDE_MARGIN` を上げ `_MIN_ROI_FRAC` を下げて wide 寄りに |
| コートが縦長で pan 主体 | `max_pan_frac` を上げ `pan_alpha` を調整 |
| ボール検出が不安定 | `_W_BALL_PREDICTED` を下げ選手集団主体に |

クロップ／レンダリング出力（Phase 5.x）では、ROI を実際に切り出し
output_aspect でリサイズした仮想カメラ映像を書き出す。本フェーズの ROI 座標は
そのまま流用できる。
