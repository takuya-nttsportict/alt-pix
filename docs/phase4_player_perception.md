# Phase 4（再定義）: コート上選手の知覚 — 技術選定

> プリンシパル 3・4 に基づく記録。Phase 3 評価とユーザーとの議論を経て、
> 旧 Phase 4（背番号 OCR）・旧 Phase 5（選手トラッキング高度化）を見直し、
> **フレーミングと試合分析の共通知覚コア**として再定義した。

## 背景と問題の再定義

### きっかけ

- 当初計画では Phase 4 = 背番号 OCR、Phase 5 = ReID による選手トラッキング高度化。
- しかし最終目標（フレーミング）から逆算すると **個人識別は不要**であり、
  さらに「フィールド選手／控え／審判の区別」は OCR でも ReID でも解けないことが
  議論で明確になった。
- 加えて、本フレームワークは将来 **試合分析（アナリティクス）** にも使う方針。
  フレーミングとは処理量・レイテンシ・出力が別だが、**知覚コアは共通**。

### 二つのプロダクトと共通コア

```
        ┌─────────────────────────────────────┐
        │   共通知覚コア (Perception Core)      │
        │  ・ボール検出 (TrackNet / Phase 3)    │
        │  ・人物検出・トラッキング (MOT)        │
        │  ・コートキャリブレーション (Phase 2) │
        │  ・チーム分類・役割フィルタ           │
        └──────────────┬──────────────────────┘
                       │ 構造化出力 (JSONL: 位置・track_id・team・role)
          ┌────────────┴────────────┐
          ▼                         ▼
  ┌───────────────┐         ┌──────────────────┐
  │ フレーミング   │         │ 試合分析          │
  │ 低遅延・追従    │         │ バッチ・高精度    │
  │ 個人識別 不要   │         │ 個人識別 必要 ★   │
  └───────────────┘         └──────────────────┘
```

**背番号 OCR の位置づけ**: フレーミングには不要だが、試合分析では個人識別に必須。
よって **キャンセルではなく、試合分析フェーズ（Phase 6）まで保留**とする。

## 技術選定（プリンシパル 4：候補比較）

MOT17/Duke/SoccerNet/SVHN 学習済み重みは商用不可）。

### 1. 多物体トラッキング（MOT）

| 候補 | ライセンス | ReID 要否 | 判定 |
|---|---|---|---|
| **roboflow/trackers（採用済）** | Apache 2.0 | 内蔵（ByteTrack/OC-SORT/BoT-SORT） | **採用継続** |
| Norfair | BSD-3-Clause | 不要 | 軽量代替として保持 |
| ByteTrack / OC-SORT / BoT-SORT | MIT | 動きベースは不要 | roboflow/trackers 経由で利用 |
| StrongSORT | GPL-3.0 | 要 | ✗ コピーレフト |
| BoxMOT | AGPL-3.0 | — | ✗ 除外 |

→ **既存の roboflow/trackers（Apache 2.0）で十分**。新規導入不要。

### 2. チーム分類（チーム割当）

> **2026-06 改訂（評価 #3 を受けて）**: 当初は「ユニフォーム色クラスタリング
> （SigLIP→KMeans）」を唯一の方式としていたが、実映像で破綻した（チームが
> ぐちゃぐちゃに混ざる）。原因と再設計を以下に記録する。

**何を根拠にチームを割り当てるかは競技で異なる**（プリンシパル4）。競技種別は
入力（`--sport`）で与えられる前提とし、種別ごとに割当戦略を差し替える。

| 競技タイプ | 採用戦略 | 根拠 |
|---|---|---|
| **ネット競技**（バレー・卓球・テニス等） | **コート左右ハーフ（ネット境界）** | ネットはラリー中の物理的ハード境界。両チームは相手コートに侵入しない。足元を既存ホモグラフィで court 座標へ射影し `u<9m/u>=9m`（中央線＝ネット）で決定。色に依存せず**決定的・warm-up 不要・追加依存ゼロ** |
| **侵入型競技**（バスケ・サッカー等） | **ユニフォーム色クラスタ（SigLIP/Lab → 2-means）** | 両チームが全面を入り混じるため空間で割れない。色で自己校正するのが妥当 |

#### なぜ色クラスタ単独はバレーで破綻したか

| 問題 | 中身 |
|---|---|
| **リベロ問題** | バレーは規則でリベロが**別色**。1 チームに 2 色 → コート上は実質「A・Aリベロ・B・Bリベロ・審判」の 4〜5 色。これに `k=2` は構造的に不一致。**ハーフ法ならリベロも同じ側＝自動で正しく束ねる** |
| 似色問題 | 紺 vs 黒など近接配色で分離不能 |
| 審判混入・warm-up/refit ドリフト | クラスタ中心がぶれる |

#### 色クラスタの色空間バックエンド（侵入型競技／フォールバック時）

| 候補 | ライセンス | 判定 |
|---|---|---|
| **SigLIP → 2-means** | コード MIT＋SigLIP Apache 2.0 | **主**。ユニフォーム埋め込み。照明・画角に頑健 |
| **Lab a\*b\* ヒストグラム → 2-means** | 自前（露出なし） | **フォールバック**。L\*（明度）を捨て色度のみ→照明変化に強い（旧 HSV から改善） |
| HSV ヒストグラム | 自前 | 軽量だが照明・影に弱い。明示指定時のみ |
| CLIP / OpenCLIP | 重みの配布条項が曖昧 | △ SigLIP を優先 |

#### チーム色の役目（降格後）

フレーミング（Phase 4/5 の目的）には「コート上に誰がいるか」だけ必要で、
**チームの恒久的同一性は不要**。色クラスタはネット競技では主役を降り、将来の
用途に限定する: (1) セット間の左右入替をまたぐ恒久チーム同一性（Phase 6 分析）、
(2) ネット際の曖昧ケースのタイブレーク。

#### 説明可能性（プリンシパル7）

各 track に **判定理由**（`team_reason` / `role_reason`）を付与し、JSONL 出力と
debug 動画に表示する。例:
- `team_reason`: `court u=4.2m <9m(net) -> side L`（ハーフ法）／
  `color d=0.12 margin=0.64 -> team0`（色法）
- `role_reason`: `off-court (foot -85px) + colour outlier (d=0.51 > thr=0.30) -> referee`

### 3. 役割区別（選手か非選手か／控え・審判）

> **2026-06 改訂（評価 #4）**: 当初は「コート幾何（on/off court）を主軸」に
> していたが、バレーでは破綻することが判明し再設計した。

#### なぜ「位置」主軸が破綻するか

| 破綻 | 中身 |
|---|---|
| **サーブで選手が場外へ** | サーバーはエンドライン後方（場外）に出る。しかも Phase 3.5 で最重要視するサーバーを `off` にしては本末転倒 |
| **主審・副審はネット中央** | 隅にいない＝位置では捕まらない（ただしフレーミング影響は小なので許容） |

#### 主軸：時間積分による「参加度」（`participation.py`）

1 ゲームを通して積分すると、選手と非選手（線審・控え・映り込み）は**どの単一
フレームに依らず**明確に分かれる:

| 特徴（累積 EMA） | 選手 | 非選手 |
|---|---|---|
| コート内滞在率 | 高い | ほぼ 0 |
| 移動量（bbox 高さ正規化） | 大きい | 小さい（静止） |
| ユニフォーム色一致（任意の第3signal） | 一致 | 外れ値 |

- **サーブ例外は時間積分が自動解決**：サーバーは試合の大半でコート内に在るので
  滞在率が高いまま → 専用サーブ検出器なしで「サーバーは選手」と扱える。
- **位置は捨てず特徴の1つ**（ハードゲートにしない）。試合が進むほど判定が安定。
- track 寿命が短いうちは判定保留（`off`）し、証拠が溜まってから確定する。

#### 非対称な入退場（fast-in / slow-out）

**コートに入るのは選手だけ**という前提から、入場と退場で判定速度を非対称にする:

- **入場（fast-in）**：いま床面に居る人は警告期間なしで即 `field`。`min_frames`
  の証拠蓄積を待たない（ユーザー指示「入れる判定はできるだけ早く」）。
- **退場（slow-out）**：床面を出ても暫くは `field` を維持。退場方向で猶予を変える:
  - **エンドライン後方（u 軸方向）**＝サーブの可能性大 → `assume_server` で**長い**猶予
    （`serve_grace_frames`、既定 ~60s）。サーバーが数十秒コート外に居ても外さない。
  - **サイドライン横（v 軸方向）**＝ボール追い等 → **短い**猶予（`grace_frames`、既定 ~5s）。

#### 退場方向の空間事前分布（zone prior）

コート 4 端点 [TL,TR,BR,BL] から、場外の人物が**どの辺の外**にいるかで役割の
事前分布が変わる（`court.off_court_zone`）:

| ゾーン | 位置 | 意味 | 役割への作用 |
|---|---|---|---|
| `endline` | TL-BL の左／TR-BR の右（エンドライン後方、サイドライン間） | **サーブゾーン** | コート履歴が無くても **`field`**（サーバーの可能性大）|
| `sideline` | TL-TR の上／BL-BR の下（サイドライン外、エンドライン間） | 審判・線審・控えの定位置 | 非選手側を補強（referee/bench）|
| `corner` | 両ラインの外（隅） | **線審がライン上に被る曖昧域** | どちらにも強制せず他信号に委ねる |
| `on` | コート内 | — | fast-in で即 `field` |

ポイントは **corner を別扱い**にしたこと。エンドライン後方を一律 `field` にすると
隅に立つ線審を誤って選手にしてしまうため、「両ラインの外（隅）」は曖昧として
参加度に判断を委ねる。サイドライン外は控え（bench）も立つので、色での bench/referee
分離はそのまま残す。

#### 試合進行判定（タイムアウト／セット間）

コート内人数の EMA（`on_court_count_ema`）が `min_active_on_court`（既定 4）を
下回ったら `game_active=False`（タイムアウト・セット切替・デッドタイム）。
フレーミングはこのフラグでカメラ動作をゲートできる（ラリー中以外は固定 等）。

#### 判定表（証拠が十分溜まった track）

判定は上から順に評価する（先に一致したものを採用）:

| 状態 | 役割 |
|---|---|
| **いま床面に居る** | `field`（即時・fast-in）|
| 退場直後（エンドライン＝assume server / サイドライン）| `field`（slow-out 猶予中）|
| **エンドライン後方ゾーン（サーブゾーン）** | `field`（履歴不要の空間事前分布）|
| 証拠不足（フレーム数 < min） | `off`（保留）|
| 参加度 ≥ 閾値 | `field`（選手）|
| 非参加 ＋ チーム色 | `bench`（控え）|
| 非参加 ＋ 色外れ値／色情報なし（サイドラインゾーンが補強）| `referee`（非選手：審判・線審・映り込みの粗バケット）|

> 色信号（`dist_map`）が無い（ハーフ法割当）場合、非選手は粗く `referee` に入る。
> フレーミングでは控え・審判とも優先度を下げるため実用上問題ない。bench/referee の
> 厳密分離が要るとき（分析）は独立した選手色モデルを追加する（後続）。

#### 候補比較（参考・据置）

| 手法 | 判定 |
|---|---|
| **時間積分の参加度（位置＋移動＋アピアランス）** | **採用（主軸）**。サーブ・センター審判に頑健 |
| チーム色クラスタの外れ値＝審判 | **採用（補助）**。bench/referee 分離に使用 |
| コート幾何のみ（on/off court） | ✗ 主軸から降格（サーブ・センター審判で破綻）|
| 検出器に審判クラスを学習 | 将来オプション（要自前アノテーション）|

> 調査結論：役割区別の turnkey 商用モデルは存在しない。SoccerNet-GSR は役割
> ラベルを持つが研究用（NDA）で商用不可。

## コート幾何の次元（2D 平面 vs 3D 立体）— 設計判断

フレーミング向けに **2D 平面ホモグラフィで十分**と判断した。根拠：

| 用途 | 必要情報 | 2D で解けるか |
|---|---|---|
| 役割フィルタ（場内/場外） | 床平面上の足元位置 | ✅ 十分 |
| 仮想カメラ ROI 算出 | 画面ピクセル範囲 | ✅ 十分（奥行き不要）|
| 選手オクルージョン解消（誰が前か） | 奥行き | ❌ だが**フレーミングは個人識別不要**なので不要 |
| ボールのネット上高さ等の計量 | metric 3D | ❌ → 試合分析（Phase 6）で必要 |

フレーミングの核心は「どのピクセル範囲をクロップするか」であり、奥行きは本質的に不要。
3D が要るのは Phase 6（試合分析）のオクルージョン解消・空間スタッツから。

**昇格パス**: バレーコート実寸は規格固定（18m×9m, ネット高 2.43m）。現行の 4 端点
ホモグラフィにメートルスケールを載せれば、**単眼カメラでも床平面（Z=0）の metric 再構成**が
可能（追加センサ不要）。`画像 → H⁻¹ → コート正規化 → ×実寸(m)`。Phase 6 で床面 PnP に
拡張し、ネット高など Z 方向を既知値として立体化する。よって Phase 4/5 は 2D、Phase 6 で
同じ 4 端点から metric 3D へ継続的に拡張する。

> 4 端点の**手動指定**自体は将来コートライン自動検出（ハフ変換／学習ベース）へ
> 差し替え可能。自動化は毎試合の手間削減策として Phase 5/6 で判断する。

### 設計上のポイント

- **チーム分類は warm-up 後に確定**：最初の `--warmup` フレームでクロップを貯めて 2-means を
  一度フィット、以後は各フレームを最近傍セントロイドへ割当。試合ごとに自己キャリブする。
- **審判は色外れ値で検出**：場内選手の「最近傍チーム色距離」分布から median/MAD でロバスト
  閾値を作り、それを超える場外者を審判とする。チームと同色の場外者は控え（bench）。
- **場外を捨てずにラベル付け**：従来は場外検出を破棄していたが、分析用途では控え・審判も
  必要。`--scene-margin` 内の人物は残し役割を付与、遠方観客のみドロップ。フレーミングには
  `role=="field"` のみを渡す。
- **依存の設計**（プリンシパル 6）：KMeans は numpy 自前実装でフォールバック経路を軽量・
  決定的に保ち、scikit-learn 依存を避けた。SigLIP（transformers）はソフト依存。

### 他スポーツへの展開方針

本フレームワークは `--sport` 引数の切り替えだけで別競技に対応できる設計になっている。
拡張ポイントを競技タイプ別に以下に整理する。

#### ネット競技（卓球・テニス・バドミントン等）

バレーと同じ `CourtHalfAssigner` がそのまま使える。変更点:

| 項目 | バレー | 他ネット競技 |
|---|---|---|
| コート実寸 | 18m × 9m（`court.py` `_COURT_W/H`） | 競技規格に合わせて変更 |
| ネット中央 `_NET_U` | 9.0m | 実寸の半分（`team_assign.py` 定数）|
| `serve_grace_frames` | ~60s | 競技によるサーブ時間に合わせて調整 |
| `min_active_on_court` | 4（最小 6人の半数） | 卓球=1、テニス=1〜2 等に変更 |

court.py の `_COURT_W` / `_COURT_H` はクラス変数ではなくモジュール定数なので、
将来的には `CourtCalibration` 初期化時に競技規格を受け取る設計に変える候補。

#### 侵入型競技（バスケットボール・サッカー等）

`ColorClusterAssigner` を使う。設計上の注意点:

| 課題 | 対応 |
|---|---|
| **選手数が多くチームが混在** | 色クラスタは問題なし。バスケ5人×2=10人、サッカー11人×2=22人 |
| **审判色が第3クラスタを形成** | 現行: 色外れ値を referee バケットに入れる（変更不要）|
| **バレーのリベロ問題は無い** | 全選手が 2 色のみ → k=2 と合致 |
| **コートが使えない競技（全面侵入）** | `CourtCalibration` は任意（`--court` を省略）。ただし game_active 判定や fast-in は使えなくなる |
| **ハーフ法が通用しない** | sport を "basketball" / "generic" にすれば自動で色法に切替わる |

`make_team_assigner` でも `sport` が volleyball 以外かつ court が None の場合は自動的に
`ColorClusterAssigner` にルーティングされる（既実装）。

#### 将来: 半侵入競技（フットサル・ハンドボール等）

コートのハーフを使いたいが、GK など常に自陣に留まる選手がいる。ハーフ法が
基本成立するがエッジケース対策が必要。`CourtHalfAssigner` に
「ハーフ境界近くのトラックは色でタイブレーク」のオプションを追加することで対応できる。

## 普遍性の担保（入力条件変化への耐性）

ユーザー懸念（画角・サイズが変わり得る）への対応：

- **役割区別はコート幾何で解く** → キャリブし直すだけで成立。映像依存の学習を避ける。
- **チーム分類は埋め込み＋クラスタリング** → 色・画角に対して相対的、特定映像に過学習しない。
- **検出器・OCR の重みは自前データ／汎用バックボーン**で学習 → 配布条項と汎化の両立。
