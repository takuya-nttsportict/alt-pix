# CLAUDE.md

このファイルは alt-pix プロジェクトにおける開発の原則（プリンシパル）を定義する。
Claude は本プロジェクトで作業する際、**必ず**この原則に従うこと。

---

## プロジェクト概要

バレーボール映像（SRT/RTMP ライブ配信および MP4 バッチ）から、ボール位置と
選手（背番号単位）をトラッキングするシステム。最終目標は映像フレーミング
（自動ズーム・カメラ追従）への活用。GPU 前提。

---

## 開発プリンシパル

### 1. フェーズ駆動開発

- 開発は**フェーズ単位**で進める。
- **各フェーズごとにプルリクエスト（PR）を作成し、レビュー・マージしてから次へ進む。**
- フェーズの作業は専用ブランチで行い、`main` へは PR 経由でのみマージする。
- **新しいフェーズのブランチは、必ず最新の `main`（前フェーズがマージ済み）から分岐する。**
  - 前フェーズが未マージのまま `main` から分岐すると成果物が欠落する。これを厳禁とする。

### 2. フェーズごとの評価とフィードバック

- **各フェーズの完了時に必ず評価を行う。**1 フェーズ ＝ 1 評価ポイントとなるよう設計する。
- 評価は再現可能な手順（評価スクリプト・指標）とともに提示する。
- 評価結果に基づき方針をアップデートする。

### 3. 方針アップデート ＝ 後続フェーズの見直し

- 評価で得た知見は、**後続フェーズの計画見直し**として反映する。
- 方針変更は記録し、なぜ変えたのかを残す。

### 4. 技術選定

- **実装可能なすべての技術を俎上に並べ、比較したうえでリーズナブルなものを選ぶ。**
  - 「思いついたものをとりあえず試す」カットアンドトライ的な技術選定を禁止する。
  - 選定時は候補一覧・各々の長所短所・選定理由を文書化する。
- **Claude の独断で、合意済みの技術選定を勝手に変更しない。**
  - 変更が必要と考える場合は、根拠を示してユーザーに諮る。

### 5. ライセンス・商用利用

- 選定する技術は**商用利用可能**なものとする。
- ロイヤリティが必要なものは都度判断するが、**よほどのことがない限りオープンソースの代替技術を使う。**
  - 例：Ultralytics YOLO（AGPL-3.0）は採用しない → YOLOX（Apache 2.0）等を使う。
- 各依存ライブラリのライセンスを把握し、記録する。

### 6. 依存関係の事前設計

- **Python の依存関係問題をカットアンドトライで解決しない。**
- 採用前に、各パッケージの依存（特に numpy ABI、CUDA バージョン、torch との整合）を
  調査し、競合しない組み合わせを設計してから実装する。
- 環境は Dockerfile で固め、再現性を担保する。

### 7. デバッガビリティ

- **すべてのシステムで、ログと処理パフォーマンス表示を考慮した実装にする。**
  - 標準のログレベル（DEBUG/INFO/WARNING/ERROR）を実装する。
  - DEBUG: フレーム単位の生値。INFO: 区間サマリ（fps、検出率等）。
    WARNING: 回復可能な異常。ERROR: 致命的エラー。
  - 処理時間・fps・各段の検出数など、性能と挙動を観測できる出力を必ず用意する。

## ブランチ運用

| ブランチ | 用途 |
|---|---|
| `main` | 各フェーズのマージ先。常に評価済み・動作する状態を保つ。 |
| `claude/phaseN-*` | フェーズ N の作業ブランチ。PR 経由で `main` にマージ。 |

- 作業開始時に `git fetch origin && git checkout -b claude/phaseN-xxx origin/main` で
  最新 main から分岐する。

## フェーズ計画（現時点）

> 評価結果に応じて後続フェーズは見直す（プリンシパル 2・3）。
>
> **2026-06 改訂**: フレーミングからの逆算と、将来の試合分析（アナリティクス）
> 用途を踏まえ、旧 Phase 4（背番号 OCR）・旧 Phase 5（選手トラッキング高度化）を
> **「コート上選手の知覚」共通コア**として再定義した。フレーミングには個人識別が
> 不要なこと、役割区別（フィールド/控え/審判）は OCR でも ReID でも解けず
> コート幾何＋色クラスタで解くのが普遍的なこと、が決め手。背番号 OCR は
> 試合分析に必要なため**キャンセルではなく Phase 6 まで保留**。
> 詳細・候補比較は `docs/phase4_player_perception.md`。

- **Phase 1**: 基盤（ストリーム受信・人物検出・初期トラッキング） … マージ済み
- **Phase 2**: ボール検出・コートキャリブレーション・ロギング整備 … 評価・マージ調整中
- **Phase 3**: ボール検出精度の改善（最優先） … 評価完了・PR 中
- **Phase 3.5**: サーブ起点・低速ボールのトラッキング対策 … フレーミング実装後に
  実データで影響を測定してから対策選定（失敗モードは後述）
- **Phase 4（再定義）**: コート上選手の知覚（人物 MOT＋チーム分類＋役割フィルタ）。
  フレーミング・試合分析の**共通知覚コア**。個人識別は不要
- **Phase 5**: フレーミング統合（ボール＋選手集団から ROI 算出・カメラ追従）
- **Phase 6（プロダクトレベル）**: 試合分析（tracklet 単位の背番号 OCR・個人
  trajectory・スタッツ）。ここで OCR の保留を解除。個人識別が必要

## Phase 3 評価所見と既知の失敗モード

WASB-SBDT の TrackNetV2（volleyball 事前学習）を、3840×800 のウルトラワイド
パノラマ映像に対し、ネイティブ高さの 16:9 重複タイル（4枚）で評価した。
ヒートマップピークは明確な**二峰性**（confident-or-blind）を示し、conf 閾値は
谷の 0.5 が最適。生検出率は約 33%。見失いの内訳から、以下 3 つの失敗モードを特定した。

| モード | 状況 | ヒートマップ | 原因（仮説） | 現対策 | 後続対策 |
|---|---|---|---|---|---|
| **1. 低速・静止** | サーブ前の床ドリブル | うっすら反応（0.1〜0.4） | TrackNetV2 は 3 フレーム間の動きを手掛かりとするため、ほぼ静止のボールは時系列パターンが出ない。学習データの偏りも疑われる | なし（閾値を下げると偽陽性増） | **Phase 3.5 で対策（最重要）** |
| **2. 高速スパイク** | アタックで対角へ | ほぼゼロ（<0.1） | 大きな移動量＋モーションブラーで「3 フレームの軌跡」が崩れる。アーキテクチャ上の限界 | Kalman 外挿で橋渡し（MAX_MISS_FRAMES=25） | Phase 6 で再検討 |
| **3. 高い軌道** | 高く上がったボール | ほぼゼロ | タイル境界での分断、または画面上端付近で学習データ外の可能性 | Kalman 外挿で橋渡し | Phase 3.5 調査に含める |

### Phase 3.5: サーブ起点・低速ボール対策（計画）

**動機**: サーブはバレーのプレー起点であり、ここからボールを追えることは
フレーミング用途で最重要。モード 1（サーブ前ドリブル〜サーブ初動）を確実に
捕捉する必要がある。Phase 3 では現アプローチの限界として記録し、本フェーズで対策する。

**候補アプローチ（プリンシパル 4 に従い比較してから選定）**:

1. **静止・低速ボール用の補完検出器**
   - 円形ホフ変換／テンプレートマッチ／背景差分など古典 CV。ボールが床付近で
     ほぼ静止する局面に限定適用。商用可・依存軽量だが誤検出対策が要る。
2. **TrackNet のファインチューン／再学習**
   - 自前のバレー映像（サーブ前後）にアノテーションを付け、低速ボールを学習させる。
     根本対策だがデータ整備コストが高い。ライセンス（WASB=MIT）は再学習可。
3. **時系列の状態機械でサーブ局面を検出し閾値・探索を切替**
   - 「サーバー静止 → トス → 打球」を選手姿勢やボール候補の文脈から推定し、
     その局面だけ conf を下げる／関心領域を絞る。偽陽性を局所化できる。
4. **小物体検出器（YOLOX-s 等）のボール特化ファインチューン併用**
   - 動きに依存しない単フレーム検出で静止ボールを補完。Phase 2 の検討資産を活用。
     Ultralytics は不可（AGPL）。YOLOX（Apache 2.0）で実施。

**選定方針**: 上記を候補一覧・長所短所・選定理由とともに文書化し、ユーザー合意の
うえ着手する（プリンシパル 4・5）。データ整備が必要な 2・4 は工数を見積もる。

## 技術スタック（合意済み）

| 領域 | 採用技術 | ライセンス |
|---|---|---|
| ストリーム受信 | PyAV | LGPL |
| 人物検出 | YOLOX (ONNX Runtime) | Apache 2.0 |
| 選手トラッキング | roboflow/trackers (ByteTrack 等) | Apache 2.0 |
| ボール検出 | TrackNetV2 (WASB-SBDT) | MIT |
| チーム分類 | SigLIP → KMeans (roboflow/sports 流) | MIT＋Apache 2.0 |
| 役割区別 | コート幾何＋色クラスタ外れ値（自前） | - |
| 姿勢推定（任意） | mmpose / RTMPose | Apache 2.0 |
| 背番号 OCR（Phase 6 保留） | EasyOCR / PARSeq | Apache 2.0 |
| 実行環境 | Docker (nvcr.io/nvidia/pytorch) | - |

**ライセンス除外確定**: Ultralytics YOLO（AGPL）、BoxMOT（AGPL）、StrongSORT（GPL）、
SportsLabKit（GPL）、volleyball_analytics（GPL）、VolleyVision（無ライセンス＋AGPL 依存）。
**重みの注意**: コードが許諾的でも MOT17/Duke/SoccerNet/SVHN 学習済み重みは商用不可。
検出器・ReID・OCR は自前データ／汎用バックボーン（ImageNet・LUPerson・SOLIDER 系）で学習する。

技術選定を変更・追加する場合は、本表とプリンシパル 4・5 に従って候補比較を行うこと。
Phase 4 以降の選定根拠は `docs/phase4_player_perception.md` を参照。
